# Chương 4.2: Khái niệm Table Formats (Lakehouse) - Phép màu trên mặt Hồ Dữ liệu

**Mục tiêu bài viết:** Giúp Kỹ sư (Mid-level) hiểu rạch ròi giới hạn của Data Lake truyền thống. Nắm vững cơ chế hoạt động của lớp Metadata (Table Format) để mang sức mạnh của Database (ACID, Update/Delete) lên Object Storage. Phân tích sâu sắc sự đánh đổi giữa hai chuẩn lưu trữ: Copy-on-Write và Merge-on-Read.

---

## I. Mindset Shift: Từ "Đầm lầy" (Data Swamp) lên "Biệt thự" (Lakehouse)

### 1. Nỗi đau của Data Lake thuần túy

Data Lake thế hệ 1 (thuần túy dùng Hadoop HDFS hoặc S3/MinIO để chứa hàng triệu file Parquet/ORC) từng là bến đỗ mơ ước của mọi doanh nghiệp vì chi phí lưu trữ cực rẻ. Tuy nhiên, khi hệ thống phình to lên mức Petabytes, những kẽ hở chí mạng bắt đầu xuất hiện đẩy hệ thống từ "Hồ dữ liệu" thành một **"Đầm lầy dữ liệu" (Data Swamp)** không thể kiểm soát.

**a. Bản chất Bất biến (Immutable) của file Parquet trên S3**
Dữ liệu trên Object Storage (S3, MinIO) tuân theo một đạo luật vật lý nghiêm ngặt: **Read-Only / Append-Only (Chỉ đọc hoặc Ghi thêm)**. 
- Giả sử bạn có một file `sales-2023.parquet` nặng 1GB chứa 10 triệu giao dịch. Đột nhiên phát hiện giao dịch của user khách hàng A bị sai một số 0.
- **Bi kịch:** Bạn **KHÔNG THỂ** mở cái file Parquet 1GB đó ra, sửa trực tiếp 1 dòng bị sai (như thao tác UPDATE trong MySQL), rồi lưu đè nhè nhẹ được. S3 không cho phép bạn luồn lách cập nhật từng byte. 

**b. Cơn ác mộng Tuân thủ (GDPR Right to be Forgotten)**
Theo Luật bảo mật GDPR của Châu Âu, nếu một người dùng yêu cầu: *"Hãy xóa vĩnh viễn toàn bộ dữ liệu của tôi khỏi hệ thống của các anh ngay lập tức!"*, bạn phải làm gì trên Data Lake thuần túy?
- Lập trình viên phải hì hục viết một Job **Apache Spark** khổng lồ.
- Job này phải lôi **toàn bộ** thư mục chứa hàng ngàn file Parquet lên bộ nhớ RAM.
- Chạy lệnh `FILTER (user_id != 'khach-hang-A')` để vớt hết tất cả mọi người (trừ user kia ra).
- Lưu đè lại **Toàn bộ** đống rác dữ liệu đó xuống S3 bằng các file Parquet hoàn toàn mới, rồi trân trối ngồi xóa thủ công đống file cũ đi.
- *Hậu quả:* Một thao tác lẽ ra chỉ tốn 0.1 giây trên PostgreSQL giờ đây thiêu rụi hàng chục giờ tính toán cực nhọc của Spark và tiêu tốn hàng đống tiền điện/I/O. Đây là thiết kế "hủy diệt", cực kỳ tốn kém và nguy hiểm.

**c. Rủi ro "Dirty Read" (Thiếu tính toàn vẹn ACID)**
*ACID (Atomicity, Consistency, Isolation, Durability)* là chén thánh của cơ sở dữ liệu giao dịch (RDBMS). Data Lake nguyên thủy không hề bảo kê các nguyên tắc này.
- Giả sử Spark Job đang hì hục chạy cái lệnh Ghi đè (Overwrite) file rác lên S3 ở ví dụ trên. Nó đang trong quá trình xóa file cũ và ghi dở dang file mới (chưa xong).
- Xui xẻo thay, đúng lúc đó một Data Analyst dùng **StarRocks** hoặc **Trino** nhảy vào SELECT bảng đó để làm báo cáo cho Giám đốc. 
- *Bom nổ:* StarRocks sẽ vớ bừa những cái data nửa nạc nửa mỡ (một nửa của file cũ chưa xóa hết, một phần của file mới đang ghi dở). Hệ thống hoặc là **văng lỗi tung tóe (File Not Found)**, hoặc kinh dị hơn: **Trả về một con số thống kê Doanh thu hoàn toàn sai xì cọt (Dirty Read)**. Sếp nhìn báo cáo và đưa ra quyết định sai lầm.

Những nỗi đau thảm khốc này chính là phát súng mở màn, buộc ngành Data Engineering phải khai sinh ra một khái niệm vĩ đại nối liền khoảng cách giữa Data Warehouse và Data Lake: Sự ra đời của **Lakehouse Architecture (Kiến trúc Biệt thự trên mặt hồ)**.

### 2. Sự ra đời của Open Table Formats (Iceberg, Hudi, Delta)

Để giải quyết các bài toán ACID, Update/Delete trên một bề mặt lưu trữ bất biến như S3, các kỹ sư từ Netflix, Uber và Databricks đã tạo ra một lớp công nghệ mới gọi là **Open Table Formats** (như Apache Iceberg, Apache Hudi, Delta Lake).

**Định nghĩa chuẩn xác:** 
Lưu ý kỹ thuật: **Table Format KHÔNG PHẢI là một định dạng file** (nó không giống như Parquet, ORC hay Avro). Phải luôn nhớ rằng ruột vật lý bên trong vẫn là hàng triệu file Parquet quen thuộc.
Về bản chất, nó là **một lớp Siêu Dữ liệu (Metadata Layer) cực kỳ thông minh** nằm đè lên trên bề mặt các thư mục chứa file Parquet đó.

**Nhiệm vụ cốt lõi:**
Nhiệm vụ duy nhất của lớp áo Metadata này là lùng sục, ghi dép và tổ chức lại vòng đời (phiên bản) của các file Parquet nhỏ lẻ bên dưới. Từ đó, nó **đánh lừa (Trick)** các engine tính toán bề trên (như Apache Spark, Trino, Flink) tin rằng:
*"Đống file Parquet lộn xộn, bất biến dưới lòng S3 kia thực ra là một **Bảng SQL (SQL Table)** hoàn hảo, có khả năng thực hiện các lệnh ACID như `INSERT`, `UPDATE`, `DELETE`, `MERGE` một cách an toàn tuyệt đối y hệt như PostgreSQL, nhưng mang khả năng mở rộng vô hạn của Cloud!"*.

## II. Giải phẫu Tính năng (Core Capabilities of Lakehouse)

Điều gì đằng sau lớp áo Metadata thần thánh kia khiến các định dạng như Iceberg, Hudi hay Delta Lake làm chao đảo thế giới Data Warehouse truyền thống (như Snowflake, Redshift)? Dưới đây là 3 siêu năng lực cốt lõi.

### 1. ACID Transactions trên Object Storage

Như đã nói ở phần trước, S3/MinIO không hỗ trợ sửa file rỗng. Vậy Table Format tạo ra hiệu ứng Update/Delete an toàn (ACID) bằng cách nào? Câu trả lời là **Optimistic Concurrency Control (OCC)** và **Metadata Pointers (Con trỏ siêu dữ liệu)**.

**Cơ chế hoạt động:**
- Khi bạn chạy lệnh `UPDATE sales SET amount = 100 WHERE id = 1`, hệ thống không hề mở file Parquet cũ ra để sửa chữa (sự bất biến được giữ nguyên).
- Thay vào đó, Spark sẽ thầm lặng ghi ra một (hoặc nhiều) **file Parquet hoàn toàn mới** chứa luồng dữ liệu đã được cập nhật.
- Cùng lúc đó, nó viết ra một file Metadata mới (như một bản đồ). Bản đồ này **từ bỏ (không trỏ mũi tên)** vào file Parquet cũ nữa, mà **trỏ vạch (point)** thẳng sang file Parquet mới vừa tạo.
- **Tính Toàn vẹn (Atomicity/Isolation):** Quá trình chuyển trỏ Metadata này diễn ra trong một chớp mắt (Atomic Swap). Chỉ khi nào file Parquet mới được **ghi xong 100%** và file Metadata mới **hoàn tất**, hệ thống mới tính là thành công (Commit). 
- Nhờ vậy, nếu StarRocks có nhảy vào đọc đúng lúc Spark đang ghi dở, nó vẫn đọc file Metadata cũ $\rightarrow$ trỏ vào file Parquet cũ $\rightarrow$ lấy được dữ liệu cũ nhưng **tuyệt đối hoàn chỉnh**. Hiện tượng "Dirty Read" bị triệt tiêu hoàn toàn.

### 2. Time-Travel & Snapshot Isolation (Git cho Dữ liệu)

Vì Table Format giải quyết bài toán Update/Delete bằng cách tạo file mới và "chuyển con trỏ", chứ không lập tức ra lệnh xóa hẳn file gốc trên S3. Hành lang cơ chế này vô tình tạo ra một phép màu: **Sự tồn tại song song của nhiều Phiên bản dữ liệu (Snapshots)**. Bạn có thể hiểu hệ thống này hoạt động y xì đúc như **Git commit** trong lập trình phần mềm.

**Kịch bản giải cứu:**
- Hệ thống luôn lưu giữ lịch sử các thay đổi. Mỗi lần có lệnh `INSERT/UPDATE`, hệ thống sinh ra một Snapshot (Phiên bản $V_1 \rightarrow V_2 \rightarrow V_3$).
- Giả sử lúc 10h sáng, lệnh ETL chạy nhầm, xóa mất 1 triệu dòng dữ liệu khách hàng (Snapshot $V_{10}$). Sếp đang làm báo cáo thì nhảy dựng lên. 
- *Phép màu:* Là một Data Engineer, bạn không cần phải lục tung log hay chạy lại data từ đầu nguồn. Bạn chỉ cần gõ một câu SQL kinh điển gọi là **Time-Travel (Du hành thời gian)**:
  ```sql
  SELECT * FROM sales FOR SYSTEM_TIME AS OF '2026-03-05 09:00:00'
  ```
- Lập tức, lớp Metadata sẽ dở sổ Nam Tào, lật ngược trở lại quá khứ, trỏ thẳng vào các file Parquet đang nằm im lặng trên S3 chưa bị xóa ở thời điểm 9h sáng. Dữ liệu quay trở lại nguyên vẹn và báo cáo của Sếp lên số chính xác. Cơ chế này cực kỳ hữu dụng để Rollback/Khôi phục thảm họa (Disaster Recovery) khi Pipeline bị sập.

### 3. Schema Evolution (Tiến hóa cấu trúc an toàn)

Bạn còn nhớ nỗi đau của Data Lake nguyên thủy không? Khi Backend đòi thêm cột hay đổi một cột từ `Integer` sang `BigInt`, Data Engineer phải hì hục viết job **Rewrite (ghi đè)** lại hàng triệu file Parquet cũ, mất nửa ngày trời.

**Với Table Formats, điều này biến mất!**
- Thay vì ghi chết tên cột (Ví dụ cột A tên là `user_id`) vào thẳng trong lòng file Parquet như trước đây. Table Formats (đặc biệt là Iceberg) sẽ **đánh mã ID (ID Tracker)** cho từng cột (Ví dụ: Column ID = 1 $\rightarrow$ Tên là `user_id`). 
- Khi có yêu cầu đổi tên cột thành `customer_id`, bạn chỉ chạy một lệnh duy nhất:
  ```sql
  ALTER TABLE sales RENAME COLUMN user_id TO customer_id
  ```
- Máy tính chỉ việc chui vào file Metadata nhỏ xíu, đổi dòng `Column ID = 1 -> customer_id`. 
- **Kết quả thu được:** Bạn cập nhật Schema cho hàng Petabytes dữ liệu (với hàng chục ngàn file Parquet bất biến dưới đáy hồ) chỉ mất **chưa tới 1 giây** mà hoàn toàn **không cần Rewrite (No file rewrite needed)** bất kỳ một file vật lý nào cả. Dữ liệu từ đó về sau tiếp tục trôi chảy êm ái. Mọi thứ được tiến hóa cực kỳ an toàn.

## III. Cuộc chiến Lưu trữ: Copy-on-Write (CoW) vs. Merge-on-Read (MoR)

Đây chính là phần "ăn tiền" nhất của chương Table Formats. Việc hiểu sâu sắc cơ chế ghi âm thầm phía dưới S3 này sẽ quyết định việc bạn cấu hình Apache Hudi (hoặc Iceberg) tối ưu đến mức nào cho từng loại Data Pipeline cụ thể. Hai triết lý kiến trúc kinh điển nhất là **Copy-on-Write (CoW)** và **Merge-on-Read (MoR)**.

### 1. Copy-on-Write (CoW) - Tối ưu cho Đọc Phân tích

Đây là phương thức nguyên thủy và an toàn nhất, đặt trọng tâm vào việc **phục vụ truy vấn đoạt tốc độ tối đa**.

**Cơ chế hoạt động:**
- Giả sử hệ thống đang có một file Parquet gốc dung lượng 100MB (chứa nén khoảng 1 triệu dòng).
- Khi có **đúng 1 dòng** Update rót vào, toàn bộ tiến trình Ghi sẽ dừng lại để làm nhiệm vụ "Bản sao lưu".
- Hệ thống sẽ cấp phát RAM, **đọc lại toàn trọn vẹn file Parquet cũ (100MB) kia lên**, tìm đến dòng chứa dữ liệu cũ cần sửa. 
- Nó trực tiếp thay thế dữ liệu thành dòng mới trên RAM. Sau đó **ghi ngược toàn bộ 1 triệu dòng (đã Update) ra một file Parquet hoàn toàn mới**. File 100MB cũ lúc này bị đánh dấu trong Metadata là "đã xóa" (chờ dọn rác).

**Điểm mạnh (Strengths):**
- **Tốc độ truy vấn (Read) cực kỳ nhanh (như Data Lake bình thường):** Dữ liệu trên ổ cứng lúc nào cũng ở trạng thái ngăn nắp, gọn gàng, file to, chuẩn Parquet nguyên chất 100%. Engine phân tích khi Select không cần tốn bất kỳ chu kỳ điện toán thừa thãi nào râu ria, chọc thẳng vào là đọc vù vù với đầy đủ sức mạnh Columnar và Data Skipping.

**Điểm yếu (Weaknesses):**
- **Write Amplification (Khuếch đại ghi):** Hãy tưởng tượng, chỉ vì muốn sửa đổi đúng **1 Byte** dữ liệu (Ví dụ đổi tuổi từ `25` thành `26`), hệ thống nhẫn tâm bắt bạn phải trả chi phí I/O để đọc ra, rồi lại ghi lại nguyên một file siêu bự hàng trăm MB. Sự cồng kềnh này khiến CoW trở nên thao tác **cực kỳ chậm** và biến thành thảm họa nếu áp dụng cho những luồng **Streaming/CDC** bắn dữ liệu Update liên tục hàng nghìn lần mỗi giây.

### 2. Merge-on-Read (MoR) - Tối ưu cho Ghi Liên tục

Để khắc phục nhược điểm "Khuếch đại Ghi" lười biếng của CoW, chuẩn **Merge-on-Read** ra đời với mệnh lệnh: *"Chỉ vứt dữ liệu update vào đó thật nhanh, khi nào thằng nào muốn đọc thì tự đi mà ghép lại!"*.

**Cơ chế hoạt động (Ghi siêu tốc):**
- Khi có 1 dòng Update bắn vào, hệ thống **KHÔNG HỀ ĐỤNG VÀO** mớ file Parquet gốc khổng lồ.
- Thay vào đó, thao tác sửa đổi này lập tức được ghi chép vào một **File Log** dung lượng cực nhỏ (thường là định dạng Hàng - Row-based như **Avro**). File log này nằm song song với file Parquet. Thao tác Ghi kết thúc êm ái tại đây.

**Nút thắt - Khi nào gộp? (On-the-fly Merge & Compaction):**
- **Lúc Đọc (Merge-on-Read):** Khi Data Analyst (user) gõ câu lệnh truy vấn Read. Engine lúc này (như Trino/StarRocks) phải gánh một trọng trách tính toán nặng nề. Nó sẽ phải tự load cái file Parquet "Cũ" lên, lôi cả file Avro Log "Mới" lên bộ nhớ RAM. Sau đó, tiến hành ép nén **MERGE hai file này vào nhau ngay trên RAM** để cho ra kết quả cuối cùng phản ánh đúng 1 dòng dữ liệu đã update đó.
- **Dọn dẹp ngầm (Compaction):** Tất nhiên, nếu không làm gì thì dồn một chặp file Avro Log sẽ to bằng cái đình. Do đó, một tiến trình ngầm (Compaction Job) sẽ được cấu hình (ví dụ chạy lúc nửa đêm). Job này chạy âm thầm phía sau, lấy toàn bộ file log ghép vào file Parquet gốc, và cuối cùng tạo thành một file Parquet "ngừa" chuẩn xác.

**Điểm mạnh (Strengths):**
- **Tốc độ Ghi (Write) siêu tốc, độ trễ cực thấp:** Thao tác nhét từng dòng update vào file Avro khiến hệ thống nhẹ bẫng. Mô hình này sinh ra hoàn hảo **để phục vụ (Target) các luồng Ingestion CDC từ Debezium/Kafka** có tần suất bắn update liên tục với độ trễ (Latency) gần nhịp Real-time nhất.

**Điểm yếu (Weaknesses):**
- **Tốc độ truy vấn (Read) bị chậm đi:** Tốc độ phản hồi lệnh Select bị kéo tụt lại do mất một khoảng chi phí (Overhead) để vi xử lý Engine phải hì hục thực hiện phép cộng gộp (Merge-on-the-fly) hai định dạng khác nhau ngay tại thời điểm vắt sữa dữ liệu (Read-time).

## IV. Thực chiến: Apache Hudi vs. Apache Iceberg

Nếu dạo quanh các diễn đàn Data Engineering, bạn sẽ thấy cuộc chiến giữa Hudi, Iceberg và Delta Lake nổ ra liên tục. Tuy nhiên, việc so sánh tính năng từng ly từng tí (Feature-by-feature) là vỗ nghĩa lúc này vì các dự án mã nguồn mở đang "sao chép" (Copy) ý tưởng của nhau từng ngày. 

Để thể hiện tầm nhìn kiến trúc sư (Architect Mindset), bạn KHÔNG CHỈ so sánh tính năng. Bạn cần nhìn vào **Triết lý Thiết kế Cốt lõi (Design Philosophy)** và xuất thân của chúng.

### 1. Apache Hudi: Quái vật của Streaming, Incremental và Update

**Xuất thân:** Hudi ra đời từ trong lòng của **Uber** - công ty gọi xe hàng đầu thế giới mưu cầu xử lý định vị hàng tỷ cuốc xe, với trạng thái cuốc (Booking Status) thay đổi từng giây một.

**Triết lý cốt lõi:**
Uber sinh ra Hudi không phải để làm Data Lake tĩnh, mà thiết kế nó như một cỗ máy **Incremental Processing (Xử lý dữ liệu gia tăng)**. Nó cực kỳ căm ghét việc phải viết lại (Rewrite) file quá to.

**Điểm mạnh thực chiến (Thế độc tôn):**
Hudi tập trung sinh lực vào việc hoàn thiện cỗ máy **Merge-on-Read (MoR)**, cơ chế **Upsert** (Cập nhật nếu có, tạo mới nếu chưa có) và luồng xử lý Incremental Pull. 
Sinh ra để tích hợp sâu và "chơi chung" cực tốt với hệ sinh thái bám sát Real-time như **Apache Kafka, Spark Streaming** hay luồng bắt sự kiện **CDC**.
*Ứng dụng:* Nếu công ty bạn làm E-Commerce, Logistics, hay Ride-Hailing, có hàng trăm Database MySQL chửi xả Update xuống S3 liên tục mỗi giây và bạn cần độ trễ dữ liệu siêu thấp (Low Latency) $\rightarrow$ Hãy chọn Hudi.

### 2. Apache Iceberg: Kiệt tác Tinh tế về Siêu dữ liệu

**Xuất thân:** Iceberg là đứa con cưng sinh ra từ nội bộ **Netflix** - Đế chế Streaming phim khổng lồ, nơi họ đang nằm đè lên những bảng dữ liệu có dung lượng Petabytes (Hàng chục triệu file Parquet trong một bảng).

**Triết lý cốt lõi:**
Netflix không gặp quá nhiều khó khăn về Streaming hay Update liên tục như Uber (vì phim cứ thêm vào là xem). Nỗi đau chí mạng của Netflix là: **Sự sụp đổ của Hệ thống quản lý File (Hive Metastore / DFS NameNode)**.
Khi cái thư mục S3 chứa đến hàng chục vạn (100,000+) file Parquet, lệnh gõ `SELECT` đơn giản khiến Data Warehouse phải scan lật từng file một. NameNode lăn ra chết ngắc. Iceberg được sinh ra với một kiến trúc cực kỳ **Thanh lịch (Elegant design)** để khắc phục giới hạn vật lý này.

**Điểm mạnh thực chiến (Thế độc tôn):**
Sức mạnh của Iceberg nằm ở **Sổ cái siêu dữ liệu phân cấp thông minh (Manifest Files Metadata)**. Lớp Metadata của Iceberg không phải là một chiếc Database cồng kềnh (như Hive), mà nó là chuỗi nhánh cây cấu trúc Cây (Tree-structure) các file theo dõi dấu vết.
- Engine khi đọc thậm chí Không cần gọi lệnh List Directory tốn kém xuống S3. Iceberg nắn thẳng đường bay của Engine (như Trino, StarRocks, Snowflake) bỏ qua 99% lượng file dư thừa.
- Iceberg trở thành định dạng "Quốc Dân" cho thế giới Data Analytics. Hệ sinh thái kết nối (Ecosystem & Interoperability) của nó là hoàn hảo. Được support tận răng bởi hầu hết mọi cỗ máy tính toán.

## V. Sổ tay Thực chiến: Anti-patterns, Interview & Decision Framework

### 1. Anti-patterns (Sai lầm vận hành chí mạng)

**a. Ám ảnh Copy-on-Write (CoW) cho luồng CDC Real-time**
- **Sai lầm:** Dùng Hudi/Iceberg định dạng CoW để đón luồng CDC (Change Data Capture) từ Debezium (PostgreSQL) bắn Update liên tục mỗi giây.
- **Hậu quả:** Job Spark/Flink Streaming của bạn sẽ rơi vào vòng lặp chết chóc. Máy chủ chạy cạn kiệt CPU nhưng Pipeline cứ cắm đầu độ trễ (Lag) lên hàng giờ đồng hồ. Nguyên nhân do hiện tượng **Write Amplification** cực đoan: Debezium đẩy lên 10 byte thay đổi mỗi giây, nhưng Job cắm đầu hì hục cày cấu đọc lại - ghi đè hàng chục file Parquet 100MB liên tục. Tốn tiền I/O AWS vô tội vạ.

**b. Lãng quên dọn rác (Compaction) khi dùng Merge-on-Read (MoR)**
- **Sai lầm:** Công xưởng luân chuyển siêu tốc MoR khiến data chảy xuống Data Lake mượt như nhung. Data Engineer hí hửng vứt đó đi cà phê suốt 6 tháng trời mà "quên" cấu hình tiến trình gộp file (Compaction Job).
- **Hậu quả:** Vào một ngày đẹp trời tháng 6, Sếp tổng mở Dashboard StarRocks lên xem báo cáo quý. Hệ thống bị chôn cứng (Treo Cluster). Sếp chửi, bạn vào kiểm tra thì tá hỏa: Ở dưới đáy S3 đang có... 10 triệu file log Avro siêu vụn vặt tí hon sinh ra từ 6 tháng qua nằm rải rác bên cạnh file Parquet. Trino/StarRocks chết ngợp vì phải tự thân xử lý Merge toàn bộ 10 triệu file này nhét lên RAM *on-the-fly* ngay trong khoảnh khắc đó. 

### 2. Sổ tay Phỏng vấn System Design (Level Mid/Senior)

Những câu hỏi trực diện giúp người phỏng vấn bóc trần năng lực "thực chiến" của ứng viên:

**Câu 1: "Cty đang xây Data Lakehouse, luồng Ingestion nhận sự kiện CDC (rất nhiều thao tác UPDATE/DELETE) từ MySQL đẩy lên Kafka. Theo bạn, nên cấu hình Apache Hudi/Iceberg chạy bằng CoW hay MoR?"**

*Đáp án điểm 10:* "Bắt buộc phải chọn **Merge-on-Read (MoR)** ạ. Vì luồng CDC từ Kafka mang đặc thù là bắn thông báo Update/Delete vô cùng vụn vặt và liên tục. 
- Nếu dùng CoW, hệ thống sẽ sụp đổ vì chi phí Khuếch đại Ghi (Write Amplification) khổng lồ khi phải liên tục tạo lại các file Parquet to. 
- Dùng MoR sẽ giúp ghi các thay đổi cực kì nhẹ nhàng và siêu tốc độ trễ thấp vào các file **Delta Log (Avro)**. 
- Tuy nhiên, để đảm bảo ngày hôm sau Team Data Analyst truy vấn không bị chậm (do *Merge on-the-fly* quá tốn RAM), em sẽ cấu hình bắt buộc thêm một tiến trình ngầm **Compaction Job** chạy định kỳ mỗi đêm lúc 2h sáng rảnh rỗi. Job này sẽ gộp thật sự các file log đó vào Parquet, trả lại tốc độ Read siêu việt bằng Parquet nguyên chất cho ngày hôm sau."

**Câu 2: "Trong Data Lake truyền thống gốc (file Parquet thường), việc đổi tên một cột (Ví dụ: `id` thành `customer_id`) là ác mộng. Tại sao lớp áo Table Formats (như Iceberg) lại làm được việc này nhẹ như lông hồng (chưa tới 1 giây) mặt dù file Parquet vẫn bất biến?"**

*Đáp án điểm 10:* "Đó là nhờ một thiết kế tuyệt đỉnh gọi là **ID Mapping** ạ.
- Trước đây hệ thống truyền thống móc nối chằng néo dữ liệu từ file cứng vào bảng bằng Tên Cột (Map by name). Vì vậy khi đổi tên, Schema dưới file vật lý bị trật khớp, buộc phải Rewrite lại toàn bộ tỉ file.
- Nhưng Table Format như Iceberg không làm thế, ở cuốn sổ cái Metadata siêu dữ liệu, mỗi cột được gán một **ID định danh duy nhất (Column ID)**. Cột `id` có ID = 1.
- Máy tính chỉ bám lấy cái ID = 1 đó để tìm vào các file Parquet. Việc đổi tên `id` $\rightarrow$ `customer_id`, hay đổi kiểu dữ liệu int $\rightarrow$ bigint, thực chất chỉ là một file chữ rất bé nằm trên file Metadata. Hệ thống chẳng cần đụng chạm đọc/ghi bất kỳ một byte vật lý nào bên dưới đáy S3. Bức tranh Schema Evolution hoàn toàn được đảm bảo an toàn cực độ".

### 3. Decision Framework (Khung thiết kế Hudi Table Type)

Tóm lại, để ghim vào đầu công thức triển khai cho mọi tình huống:

1. **Batch Ingestion (Cào data định kỳ dội lên Data Lake, Append-only):**
   - *Đặc điểm:* Chỉ nhồi dữ liệu cực lớn 1 ngày/lần. Rất ít (hoặc không có) Update/Delete.
   - *Chiến thuật:* Nhắm mắt chọn **Copy-on-Write (CoW)**. 
   - *Mục tiêu:* Tối ưu tuyệt đối tốc độ phục vụ cho Report phân tích/Dashboard Data Analyst vào ngày hôm sau.

2. **Streaming Ingestion / CDC (Luồng tốc độ cao: MySQL $\rightarrow$ Kafka $\rightarrow$ Flink $\rightarrow$ Hudi):**
   - *Đặc điểm:* Khát khao độ trễ tính bằng phút (Low Latency). Dòng dữ liệu đổ tới dồn dập mang theo Update/Delete/Upsert liên hoàn.
   - *Chiến thuật:* Bắt buộc chọn **Merge-on-Read (MoR) kết hợp Compaction**. 
   - *Mục tiêu:* Vừa cứu sống phần hạ tầng Ingestion (Ghi dứt khoát không kẹt), vừa chấp nhận hi sinh Tốc độ Truy vấn (Query) bị chậm đi một chút xíu trước khi chốt đơn bằng Job Compaction.
