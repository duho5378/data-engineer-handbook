# Chương 4.3: Object Storage vs File System - Kỷ nguyên Lưu trữ Đám mây và Sự sụp đổ của HDFS

**Mục tiêu bài viết:** Thay đổi tư duy từ việc lưu trữ có cấu trúc cây (Hierarchical) sang cấu trúc phẳng (Flat Namespace). Hiểu lý do tại sao kiến trúc Hadoop/HDFS bị thay thế bởi AWS S3 và MinIO. Nắm vững bản chất của Object Storage để tránh các bẫy chết người về chi phí API và hiệu năng khi truy vấn.

---

## I. Mindset Shift: Cuộc lật đổ của Kiến trúc Tách biệt (Decoupling)

Sự thoái trào của tượng đài Hadoop Distributed File System (HDFS) và sự lên ngôi độc tôn của AWS S3 (và mã nguồn mở MinIO) không đơn thuần là một cuộc đua công nghệ thông thường. Thực chất, đây là một cuộc "cách mạng lật đổ" về triết lý thiết kế hình thành nên toàn bộ hệ sinh thái Big Data hiện đại.

### 1. Kỷ nguyên HDFS (Hadoop Distributed File System): Bị trói buộc

Hơn một thập kỷ trước, ở thời bình minh của Big Data, các công ty công nghệ phải đối mặt với một nút thắt cổ chai vật lý cực kỳ đắt đỏ: **Băng thông mạng (Network Bandwidth) quá chậm và yếu**.

**Thiết kế nguyên khối (Monolithic Architecture):**
Nguyên lý vàng thời HDFS là: *"Thay vì cõng 10TB dữ liệu khổng lồ lết qua đường truyền mạng chậm chạp để gửi đến CPU vắt kiệt tính toán, hãy gửi cục code nhỏ bé (vài KB) đến thẳng cái máy tính vật lý đang nằm ôm đống dữ liệu đó"*. 
Khái niệm này gọi là **Data Locality (Tính toán cục bộ)**.
Để làm được điều này, **Tầng Lưu trữ (Storage)** và **Tầng Tính toán (Compute)** bắt buộc phải sống chung dưới một mái nhà. Chúng bị trói chặt vào nhau bên trong cùng một chiếc máy chủ vật lý (gọi là DataNode hay Worker Node). Mỗi máy chủ vừa dùng CPU/RAM để xử lý, vừa dùng chính ổ cứng (HDD/SSD) bên trong bung của nó để lưu trữ file.

**Nỗi đau (Sự thiếu linh hoạt chí mạng):**
Sự "trói cứng" này mang đến một tấn bi kịch về giới hạn mở rộng (Scalability) của doanh nghiệp:
- Cứ sau một đợt săn sale cuối năm, công ty cạn kiệt dung lượng ổ cứng để chứa Data gốc. Tại thời điểm đó, lượng Data tính toán sút giảm, CPU hiện tại thực sự vẫn rảnh rỗi chạy chậm chạp tới mức bỏ không.
- Để tăng sức chứa (Capacity) trên cụm HDFS, Giám đốc IT **BẮT BUỘC** phải thầu nguyên một lô Server vật lý mới nguyên khối. Tức là doanh nghiệp phải nghiến răng bỏ cả đống tiền mua thêm CPU và RAM đắt đỏ vốn không hề cần thiết, chỉ để gắn thêm được vài cái ổ cứng Disk mới vào Rack.
- Ngược lại, những lúc chạy Model Machine Learning cần sức mạnh tính toán (CPU cao vút) nhưng dữ liệu thì rất bé. Bạn cũng phải nín thở mua Server nguyên khối chứa thừa mứa ổ cứng bên trong. Sự phình to vô tổ chức này dẫn đến lãng phí chục tỷ đồng, dần đưa HDFS vào sử sách.

### 2. Kỷ nguyên Object Storage (AWS S3, MinIO): Sự giải phóng

Bước sang thập kỷ thứ ba của thế kỷ 21, khi công nghệ cáp quang Datacenter bứt phá vượt bậc. Hạ tầng đã chứng kiến **Băng thông mạng lên tới 100Gbps+ (Thậm chí 400Gbps)**. Giờ đây, lưu lượng di chuyển qua mạng nội bộ thậm chí còn nhanh hơn cả tốc độ đọc từng ổ cứng cục bộ gộp lại. Trở ngại cuối cùng cản bước việc tách rời tài nguyên đã bị phá vỡ. Đó là lúc Object Storage sinh ra để giải phóng Data Engineer.

**Thiết kế (Decoupled Compute & Storage):**
Đây là triết lý xương sống của kiến trúc Cloud hiện đại: **Tách biệt hoàn toàn Tầng Lưu trữ và Tầng Tính toán (Decoupling)**.
- **Tầng Lưu trữ (Storage):** Dữ liệu được tống khứ toàn bộ xuống một cụm lưu trữ tĩnh, rẻ rúng và cô lập (như Amazon S3 trên Cloud, hoặc ném vào cụm MinIO tự dựng tại chốn On-Premise). Cụm này câm nín làm đúng một nhiệm vụ duy nhất: Chứa hàng tỷ File vô tri dưới dạng Đối tượng (Object). Không biết làm phép toán cộng trừ nhân chia.
- **Tầng Tính toán (Compute):** Khi nào sếp yêu cầu xuất thông số báo cáo, hay User cần làm Data Science, hệ thống mới bật các cụm phân tích (Spark, Trino, Flink, Athena) lên. Các bộ xử lý đầy sức mạnh này sẽ cắm cái vòi rồng kết nối siêu phân luồng xuống S3/MinIO, bơm file băng qua đường mạng (Network) lên thẳng bộ nhớ RAM để "làm thịt".

**Lợi ích đem lại:**
- Sự tách rời vĩ đại này đem lại **Khả năng mở rộng vô hạn (Infinite Scalability) hoàn toàn độc lập với chi phí siêu rẻ**.
- Nếu Website thu thập quá nhiều Log hết chỗ chứa? Đơn giản, vô tận quẳng Log lên S3, bạn chỉ trả phí vài chục Đô-la nâng cấp dung lượng lưu trữ tủ kệ mà không cần tốn một xu xích nảy sinh mua CPU.
- Nếu Giáp Tết cần máy cày chạy Pipeline chốt báo cáo Year-End siêu nặng? Data Engineer cứ mạnh tay Spin-up (kéo lên) 100 server tính toán Cloud chạy cháy máy trong 2 tiếng đồng hồ. Xong việc tắt toàn bộ đi để tiết kiệm. Trong khi đó, hệ thống dữ liệu (S3/MinIO) vẫn bất di bất dịch nằm ngoan ngoãn chờ lần tiêu thụ sau. Cắt giảm 90% ngân sách IT của tập đoàn.

## II. Giải phẫu Kiến trúc: File System vs. Object Storage

Vậy bản chất cái Hố đen S3 kia khác gì với ổ đĩa C: chứa Windows hoặc ổ D: chứa phim trên laptop của bạn? Tại sao Amazon không mua hàng triệu cái ổ NTFS/ext4 rồi gắn vào Cloud cho chúng ta xài? Câu trả lời nằm ở "Cấu trúc quản lý".

### 1. File System (Hệ thống tập tin - POSIX)

Đây là di sản kiến trúc hình cây (Hierarchical) từ thời tiền sử của ngành Khoa học Máy tính. Linux ext4, Windows NTFS, và ngay cả HDFS đều trung thành với triết lý quản lý thư mục lồng nhau này.

**Cơ chế hoạt động (Quản lý theo Cấu trúc Cây):**
- Trong File System, dữ liệu được tổ chức theo các nhánh cây thư mục (Directory Tree). 
- Giả sử máy tính muốn lấy một file ảnh nằm sâu bên trong, hệ thống vận hành phải lần mò từng nấc một: Tứ thư mục gốc `A` $\rightarrow$ chui vào thư mục con `B` $\rightarrow$ rồi mở tiếp thư mục lớp cháu `C` $\rightarrow$ cuối cùng mới vồ lấy được file ảnh `D`.

**Giới hạn sụp đổ (Nỗi ám ảnh tràn RAM Inode):**
- Khi công ty bạn chỉ có vài chục triệu file, hệ thống cây này hoạt động cực kỳ trơn tru. 
- Nhưng Big Data không dậm chân ở đó. Khi Data Lake chứa **hàng chục tỷ file** (nhỏ giọt từ Streaming IoT hoặc Log mạng), cái cây thư mục này khổng lồ đến mức mất kiểm soát.
- Mọi con đường (Path) và cấu trúc cây (gọi là Metadata/Inode) đều buộc phải lưu giữ trên thanh RAM của máy chủ quản lý trung tâm (Ví dụ như cục NameNode của kiến trúc HDFS). Quá trình bò mẫm dạo quanh cây thư mục (Path Traversal) để tìm đúng file sẽ thiêu rụi toàn bộ sức chịu đựng của CPU. Rút cuộc, cái NameNode trung tâm bị **Tràn RAM (Out-of-Memory)** và lăn ra chết đứ đừ kéo sụp toàn bộ cụm Hadoop, dù ổ cứng chứa file bên dưới sức chứa vẫn còn mênh mông bạt ngàn.

### 2. Object Storage (Lưu trữ Đối tượng - Cấu trúc Phẳng)

Để giải quyết bài toán "Sập máy chủ quản lý trung tâm", AWS S3 (và mã nguồn mở MinIO) đã đạp đổ hoàn toàn các định luật quản lý file truyền thống. Chúng khai sinh ra kiến trúc **Cấu trúc Phẳng (Flat Namespace)**.

**Cơ chế hoạt động (Sự thật kinh hoàng: KHÔNG CÓ THƯ MỤC):**
- Trái với suy nghĩ biểu kiến của mọi kĩ sư trên giao diện Web UI, Object Storage ở dưới đáy vật lý **HOÀN TOÀN KHÔNG CÓ KHÁI NIỆM THƯ MỤC**. Thắng rẽ ngang hay đi sâu rốt cuộc đều không tồn tại khái niệm Folder lồng nhau.
- Tất cả mọi thứ (Trăm tỷ file hình ảnh, file Parquet, file Text) đều được gọi chung là các **Đối tượng (Objects)**. Và đặc biệt, tất thảy chúng đều bị vứt nằm chỏng chơ một cách hỗn độn ngay trên cùng một mặt phẳng khổng lồ vô tận.

**Cách định vị (Key-Value):**
Trong biển người vô danh đó, làm sao Spark hay Trino tìm được đúng file cần đọc?
- Nó quản lý giống y hệt một cuốn Từ điển (Key-Value Store). Mỗi Đối tượng khi quăng lên sẽ được cấp 1 cái Mã số Định danh (ID/Key) duy nhất.
- Ví dụ: `s3://my-bucket/year=2026/month=03/data.parquet`
- Đừng để giao diện dối lừa! Ở đây, `year=2026` hay `month=03` **TUYỆT ĐỐI KHÔNG PHẢI LÀ TÊN CÁC THƯ MỤC LỒNG NHAU**!. Toàn bộ cái cụm `/year=2026/month=03/data.parquet` chỉ đơn thuần đóng vai trò là một **Chuỗi Tên dài ngoằng (Tên Tiền tố - Prefix Key)** dán lên trán cái Object đó mà thôi.

**Sức mạnh hủy diệt:**
- Vì triệt tiêu hoàn toàn sự tồn tại của Mộc thư mục lồng nhau, Hệ thống trút bỏ được gánh nặng phải nuôi một cỗ máy NameNode quản lý Path Traversal lề mề. 
- Khi bạn ra lệnh truy vấn `data.parquet` trên S3, chỉ bằng 1 thao tác dò tìm ID Key **độ phức tạp O(1) (Lookup siêu tốc độ)**, hệ thống lập tức chọc thẳng và nắm đầu cái file đó lôi lên từ mặt phẳng lưu trữ.
- Kiến trúc phẳng lỳ này chính là vũ khí giúp Amazon S3 hay MinIO mang lại năng lực **Mở rộng Vô hạn (Infinite Scalability)**. Trăm tỷ hay nghìn tỷ Object cũng không bao giờ làm hệ thống bị nghẽn cổ chai hay sập RAM.

## III. Đặc tính cốt lõi của Object Storage (The S3 Standard)

Khi Data Lake quyết định "cắm rễ" lên S3/MinIO, Kỹ sư Dữ liệu phải làm quen với những luật chơi khắc nghiệt được gọi chung là "Amazon S3 Standard". Đây là bộ nguyên lý đã trở thành tiêu chuẩn vàng của ngành.

### 1. Bản chất Bất biến (Immutability)

Luật chơi số một và tàn nhẫn nhất trên Object Storage: **Mọi Đối tượng sinh ra đều là Bất biến (Immutable)**.

- Một khi bạn đã tải lên (Ghi/PUT) thành công một file Parquet dung lượng 1GB xuống S3, file đó bị "bịt kín" mãi mãi.
- Bạn tuyệt đối KHÔNG THỂ mở file đó ra, lò dò trỏ chuột vào giữa hông file để **Sửa đổi (Update/Append)** vài ba Byte hay thêm một dòng lệnh mới vào cuối file (như kiểu Ghi tiếp - Append trên thư mục máy tính thông thường).
- Nếu phát hiện file có 1 dòng ghi sai tên khách hàng? Bạn phải nhấc mông lên, cấu hình cụm máy tính tải nguyên file 1GB đó lên RAM, sửa đúng 1 dòng bị sai, sau đó hì hục Ghi đè (Overwrite) làm thành một Object 1GB lưu xuống cất lại, vĩnh biệt file cũ.
- Tới đây, chắc chắn kiến thức cũ của bạn sẽ bắt nhịp lại. Sự "bức bối/bất biến" kinh hoàng này chính là **nguồn cơn sâu xa bắt buộc Kỹ sư Data Engineer phải sử dụng đòn bẩy Table Formats (như Apache Hudi / Iceberg ở Chương 4.2)**. Thay vì để hệ thống khóc ròng vật lý trên S3, lớp áo Table Formats (với cơ chế Merge on Read hoặc Metadata Pointer) sẽ "Hack/Đánh lừa" hệ thống để chúng ta Update/Delete vô tư mà không lộ nhược điểm bất biến của Object Storage. 

### 2. Giao tiếp qua REST API (Http Transport)

Cú sốc thứ hai đến từ cách mà con người hay phần mềm nói chuyện với Object Storage.

- Nếu lập trình ứng dụng ghi file trên ổ cứng Laptop, hệ điều hành (OS) cung cấp sẵn các hàm (System Calls) để lập trình viên gọi nhẹ nhàng như `open()`, `read()`, `write()`, `close()`. Mọi thứ chạy âm thầm dính líu đến mâm từ bên trong.
- **S3 và MinIO thì không!** Bạn không thể gắn chúng thành một Ổ Đĩa (Drive) một cách thông thường. Object Storage hoàn toàn phớt lờ Hệ điều hành. Cách duy nhất để nói chuyện với nó là qua cổng thông tin mạng internet bằng giao thức **HTTP / HTTPS REST API**.
- Khi Spark hay Trino muốn đọc file, chúng thực chất đang nhấc máy chủ kích hoạt các hàm Request mạng y như cách bạn gõ URL vào trình duyệt:
  - Tải file lên? Gửi HTTP lệnh **`PUT`**.
  - Đọc tải file về? Gửi HTTP lệnh **`GET`**.
  - Xóa file? Gửi HTTP lệnh **`DELETE`**.

**Khó khăn tiềm ẩn (Network Overhead):**
Chính vì vạn vật giao tiếp dựa hoàn toàn vào việc Gọi Hàm API qua mạng, nếu cấu trúc Data Lake của bạn nát bét (ví dụ tát vào trong S3 chứa tận 1 triệu file JSON siêu nhỏ giọt thay vì nén thành file Parquet lớn), công cụ phần mềm sẽ phải gọi **hàng triệu lệnh `GET` request qua HTTP**. Trả về một kết cuộc đau thương: Nút thắt cổ chai độ trễ mạng (Network Latency) xé nát tốc độ hệ thống, và Hóa đơn tính tiền API rọi tới AWS làm sếp ngất lịm đi.

## IV. Cơn ác mộng Vận hành & Hiệu năng (Performance Nightmares)

Kiến trúc siêu phẳng mang đến sức sống vĩnh cửu, nhưng đi kèm với nó là những cạm bẫy "chết người" vô hình nếu Data Engineer vẫn ôm cái đầu mang tư duy File System truyền thống để leo lên Cloud. Hãy dè chừng 3 bóng ma sau:

### 1. Cú lừa "Rename Folder" (Ảo giác Đổi tên Thư mục)

**Thói quen trên máy tính:** Khi bạn click chuột phải đổi tên một thư mục nặng 100GB từ `Data-2023` thành `Data-2024` trên Laptop Windows, tiến trình kết thúc trong **0.001 giây**. Tại sao? Vì thư mục vật lý không di chuyển, máy tính chỉ thay đổi đúng "Con trỏ tên (Pointer)" trên bảng chỉ mục.

**Tuy nhiên, trên S3/MinIO:** Bạn nhấn lệnh "Rename Folder" thông qua công cụ CLI của AWS và mọi thứ vỡ vụn.
- Như giải phẫu ở Phần II, S3 **không có thư mục**. Cái chữ `Data-2023/` chỉ là một cái Tên Tiền tố (Prefix Key) dán dính sát sạt vào từng Object (Ví dụ: `Data-2023/file1.parquet`).
- Do đó, thao tác "Đổi tên thư mục" thực chất là hệ thống phải mở lệnh duyệt qua hàng triệu file có chứa cụm từ đó ở tên gọi. Hệ thống phải hì hục thực hiện **COPY** 1 triệu cái Object đó sang cái Object có Tên-Hoàn-Toàn-Mới (`Data-2024/file1.parquet`). Sao chép 100GB thành công xong, hệ thống ngậm ngùi quay lại **DELETE (Xóa bõ)** 1 triệu cài Object có tên cũ.
- Tùy thuộc vào băng thông và số lượng file, cú lừa siêu to khổng lồ này khiến hệ thống gồng mình chạy mất cả ngày trời, và đi kèm một cái Bill tính phí hàng triệu thao tác API PUT/DELETE đắt đỏ sấy rát túi tiền công ty.

### 2. Thắt cổ chai Prefix (API Rate Limits)

AWS S3 sinh ra được quảng cáo là "Scale tự do vô tận", nhưng khả năng điều tiết lưu lượng bắn API của nó (Rate Limit) lại có định mức phần cứng theo quy tắc.

**Giới hạn vật lý:** 
- S3 sẽ áp đặt giới hạn lưu lượng giao tiếp HTTP API tối đa cho **mỗi Cụm Tiền tố (Prefix)**. AWS hiện tại quy định mức trần là: Tối đa **3,500 lệnh PUT/COPY/POST/DELETE** hoặc **5,500 lệnh GET/HEAD** mỗi giây cho một Prefix duy nhất.

**Hậu quả khi thiết kế tồi:**
- Giả sử bạn ném toàn bộ dữ liệu cào được từ hệ thống bằng Spark hầm hố có 1000 con Worker (1000 Server trâu cày). Cả ngàn con Worker này đồng loạt xả hàng triệu file Parquet nhỏ nhẻ vào **đúng một đường dẫn lười biếng duy nhất**: `s3://bucket-chinh/raw-data/` (nghĩa là chung cái Prefix `raw-data/`).
- *Bom nổ:* Lưu lượng bắn dồn ứ hàng chục nghìn file vào một cái họng súng (Prefix) vượt mốc 3,500 PUT/s. Lập tức, mạng lưới Cloud của Amazon văng lỗi báo động đỏ khét lẹt: **`HTTP 503 Slow Down` (Từ chối dịch vụ)**.
- Quá trình Ingest Data chết gục. Pipeline chạy thất bại thảm hại. Đây là lý do sống còn mà Data Engineer phải thiết kế kỹ càng việc **Phân vùng (Partitioning)** (ví dụ: chia chẻ Prefix thành `.../year=2026/month=03/day=05/`) để "đánh lừa/rải đều" sức ép API ra nhiều mốc Prefix khác nhau, đảm bảo hệ thống nuốt trọn băng thông khổng lồ.

### 3. Small Files Problem (Kẻ thù số 1 của Data Lake)

Bất kể bạn dùng cục NameNode của lão tướng HDFS hay "Cấu trúc viễn tưởng" của Object Storage, có một kẻ thù chung kinh điển muôn đời không được lặp lại: **Cơn Bão Các File Nhỏ (Small Files Problem)**.

**Sự thật phũ phàng:**
Việc Lưu **1 triệu file có dung lượng 1KB** (Tổng 1GB) là một trải nghiệm **địa ngục tồi tệ gấp hàng trăm ngàn lần** so với việc lưu **10 file dồn nén có dung lượng 100MB** (Tổng 1GB).

**Vì sao lại sụp đổ?**
- *Khía cạnh Truy vấn (Chi phí Inode & API):* Khi Trino/Spark chui xuống Data Lake để dò tìm và lên Kế hoạch Truy vấn (Query Planning), chúng phải lật mở (GET metadata) kiểm tra bìa của từng file. Nếu đụng 1 triệu file 1KB, Engine sẽ phải lặp đi lặp lại hàm Request API qua đường truyền mạng internet 1 triệu lần cực nhọc, riêng khâu gửi nhận Request trống này thôi cũng làm chậm tốc độ báo cáo của sếp đi hàng ngàn lần. 
- *Khía cạnh Kinh tế:* Amazon S3 không tính tiền theo dung lượng file 1KB bé tí xíu, mà họ **tính tiền (Cost) trên số lượt gọi (Request) API PUT/GET**. Chúc mừng, vì lưu 1 triệu file rác 1KB, cuối tháng Công ty bạn sẽ vinh dự "Đóng thuế API" hàng ngàn Đô la cho AWS thay vì chỉ đóng 1 USD cho việc đọc 10 file 100MB sạch sẽ.

## V. Sổ tay Thực chiến: Anti-patterns, Interview & Decision Framework

Để kết thúc chương về Object Storage, dưới đây là những ngộ nhận "chết người" của các kỹ sư mới vào nghề và cẩm nang phỏng vấn System Design ăn điểm tuyệt đối.

### 1. Anti-patterns (Sai lầm phổ biến của Junior)

**Cố gắng Mount (gắn) S3 thành ổ cứng cục bộ mâm từ**
- Rất nhiều bạn sinh viên khi mới tiếp cận Cloud thấy S3 lưu trữ vô hạn rẻ quá, bèn lên mạng tìm các công cụ dối trá như `s3fs` hay `goofys` để "Mount" (gắn) cái Bucket S3 đó hóa phép thành một ổ đĩa `E:\` cục bộ trên con Server Linux.
- Sau đó, họ ngây thơ lấy ổ `E:\` đó làm chỗ cài đặt Database PostgreSQL hoặc chỗ để chạy tập tin Web động. 
- **Kết quả (Thảm họa):** Ứng dụng sập nguồn ngay lập tức hoặc treo cứng cực độ. Bất kỳ phần mềm truyền thống nào (như RDBMS) đều được thiết kế chằng néo xoay quanh một tính năng gọi là mạng lưới **Khóa Tệp tin (File Locking)** và yêu cầu độ trễ mâm từ Microseconds cực thấp. S3 là HTTP API, gọi nó qua mạng có độ trễ lớn và S3 hoàn toàn **KHÔNG HỖ TRỢ** chức năng File Locking. Dữ liệu DB sẽ hỏng hoàn toàn (Corrupted).

### 2. Sổ tay Phỏng vấn System Design (Level Mid/Senior)

Những câu hỏi hóc búa để phân loại Kỹ sư "lý thuyết" và Kỹ sư "thực chiến".

**Câu 1: "Hệ thống Spark của em đang chạy rất tốt, nhưng gần đây khi tăng tốc độ ghi file ra S3, log liên tục báo lỗi `HTTP 503 Slow Down` dù dung lượng Bucket chưa hề đầy. Dựa vào kiến trúc hệ thống, e giải thích tại sao và cách khắc phục?"**

*Đáp án điểm 10:* "Lỗi `503 Slow Down` trên S3 đa phần không đến từ hạn mức dung lượng, mà đến từ hạn mức băng thông **API Rate Limit**. Nguyên nhân là do cấu hình Spark của em đang dội quá nhiều request `PUT` vào tập trung trên **Cùng một Prefix (Đường dẫn)** cùng một lúc, vượt qua định mức phần cứng của AWS (thường là 3,500 PUT/giây/prefix). 
Cách giải quyết triệt để là phải "Tản nhiệt" phân tán lượng Request này ra. Em sẽ cấu hình lại hệ thống **Tăng số lượng Partition / Prefix** (Ví dụ chia nhỏ thư mục theo Giờ/Phút/Kênh) hoặc tốt nhất là **Bật tính năng ngẫu nhiên hóa chuỗi Prefix (Random/Hash Prefix)** để AWS rải đều file ra nhiều máy chủ lưu trữ vật lý đằng sau, hấp thụ được toàn bộ sức ép."

**Câu 2: "Có một lo ngại khi anh em mình xài Data Lake là tính nhất quán dữ liệu. Theo em, hệ sinh thái AWS S3 hiện tại đang cung cấp Tính Nhất Quán (Consistency) theo chuẩn nào?"**

*Đáp án điểm 10:* (Đa số người học tài liệu cũ vài năm trước sẽ trả lời sai câu này). 
"Dạ, kiến trúc đã có sự thay đổi lịch sử. Trước tháng 12 năm 2020, S3 sử dụng triết lý **Nhất quán cuối (Eventual Consistency)** tức là em cứ Ghi file (PUT) xong báo thành công, nhưng nửa giây sau em Đọc (GET) lại con file đó có thể chưa ra kết quả ngay (Ra List rỗng hoặc data cũ).
Nhưng nhận thấy sự bức bối của giới Data Engineer, từ 12/2020, AWS đã đập đi làm lại để S3 hỗ trợ toàn diện **Strong Read-After-Write Consistency (Nhất quán mạnh)**. Có nghĩa là ở thời điểm hiện tại: Mọi thứ File hệ thống em Ghi thành công, chắc chắn 100% lệnh truy xuất liền ngay sau đó sẽ tóm được phiên bản mới nhất. (Kèo này mã nguồn mở MinIO cũng đã hỗ trợ kiến trúc Strong Consistency trọn vẹn ạ)."

### 3. Decision Framework (Khi nào dùng loại Storage nào?)

Là Kỹ sư Dữ liệu, bạn không bao giờ dùng S3 cho mọi bài toán. Dưới đây là bộ "La bàn ra quyết định" để không bao giờ chọn nhầm hệ thống lưu trữ trên Cloud:

| Loại Storage | Độ trễ (Latency) | Chuẩn kết nối | Khả năng chia sẻ | Điểm đỗ (Use Case hoàn hảo) |
| :--- | :--- | :--- | :--- | :--- |
| **Block Storage** (Ví dụ: SSD Laptop, AWS EBS) | Siêu thấp (Chỉ vài Microseconds) | Mount cứng (Cắm cáp) | **Chỉ dính chặt vào 1 Server duy nhất** (EC2) một lúc | - Cài hệ điều hành OS (Windows/Linux OS root).<br>- Làm ổ đĩa cắm cho **Database giao dịch tốc độ cao** (Postgres, MySQL, MongoDB). |
| **File Storage** (Ví dụ: Ổ ảo mạng LAN, AWS EFS, NFS) | Thấp / Trung bình (Milliseconds) | Cấu trúc Cây Thư Mục (POSIX) | **Nhiều Server (Gắn mạng LAN)** có thể cùng đọc/ghi đồng thời. | - Ổ cứng dùng chung chứa File văn phòng của công ty.<br>- Chứa file Setting cấu hình dùng chung của hàng trăm cụm Server nhỏ. |
| **Object Storage** (Ví dụ: AWS S3, MinIO) | Trung bình (Vài chục Milliseconds) | HTTP REST API, Phẳng lỳ Không Thư mục | Mọi máy chủ trên toàn cầu đều có thể kết nối vào qua đường Link/ID. | - Vứt hình dáng Cấu trúc (Format) tùy ý.<br>- **Làm Data Lake vĩnh cửu** (Chứa Petabytes log, file Parquet, Hive).<br>- Lưu Media dạng hình ảnh, video (Youtube/Facebook dùng cái này) và Backup Archive siêu rẻ. |
