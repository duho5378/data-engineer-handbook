# Chương 8.1: Containerization - Đóng gói ứng dụng với Docker

## I. Bản chất của vấn đề: Tại sao phải dùng Container?

Trước khi đi sâu vào Docker hay Kubernetes, hãy lùi lại một bước để hiểu nỗi đau (pain point) lớn nhất của việc phát triển và triển khai phần mềm truyền thống. Một Kỹ sư dữ liệu không thể nói chuyện về Container nếu chưa nếm trải cảm giác ứng dụng chạy ngon lành trên máy cá nhân nhưng lại "chết dở" khi đưa lên Production Server.

### 1. Vấn đề môi trường (Môi trường "vỡ nát")

Trong quá khứ, các công ty thường phải sắm một chiếc máy chủ (Server/Bare metal) khổng lồ chứa hàng tá ứng dụng. Điều này đẩy các kỹ sư vào một vòng luẩn quẩn hóc búa mang tên **Xung đột phiên bản**.

*   **Xung đột thư viện (Dependencies Conflict):** Giả sử Data Team có hai dự án quan trọng đang chạy chung trên một Server vật lý. Dự án A (viết 3 năm trước) chỉ tương thích với Java 8. Trong khi Dự án B (Vừa phát triển tuần trước) lại yêu cầu nền tảng Java 17 mới nhất. Hệ điều hành trên server không thể nào cài 2 phiên bản Java global cùng một lúc mà không sinh ra rắc rối đè chéo đường dẫn (Path). Hậu quả: Để chạy được Dự án B, team phải gỡ bản Java cũ, khiến Dự án A đang chạy tốt bỗng nhiên "sập" hoàn toàn.
*   **Thiếu hụt biến môi trường (Environment Variables):** Dev A hì hục code xong một pipeline dữ liệu, test chạy trơn tru trên laptop Macbook (Môi trường Local). Nhưng khi đẩy source code đó giao cho đội Vận hành (Ops) đem lên máy chủ Ubuntu (Production) chạy thử thì báo lỗi `ModuleNotFoundError`. Lý do: Trên máy ông Dev A đã cài sẵn một vài thư viện ngầm định bằng lệnh `pip install`, hoặc ông có đặt một biến môi trường `API_KEY` ẩn trong terminal. Đội Ops hoàn toàn không biết những điều đó để cài lại y hệt trên máy chủ.
*   **Điệp khúc kinh điển:** *"Nhưng code của em vẫn chạy ngon trên máy em mà! (It works on my machine!)"* - Đây là câu nói quen thuộc minh chứng cho sự bất tương xứng môi trường.

### 2. Virtual Machine (Máy ảo - VM) vs. Container

Để giải bài toán trên, người ta từng dùng **Máy ảo (Virtual Machine - VM)**, cho đến khi **Container** xuất hiện và làm một cuộc cách mạng định hình lại toàn bộ nền IT hiện đại. Bảng so sánh dưới đây là câu hỏi nhập môn kinh điển cho vị trí DevOps/Data Engineer:

#### a. Máy ảo (Virtual Machine) - Ảo hóa phần cứng
*   **Cơ chế:** Dùng một lớp hypervisor (như VMware, VirtualBox) để chia nhỏ RAM, CPU của Máy chủ thành nhiều cục máy ảo.
*   **Đặc điểm chí mạng:** Mỗi máy ảo bắt buộc phải cõng nguyên một **Hệ điều hành Khách (Guest OS)** nặng nề (ví dụ: nguyên một cục Windows Server hay Ubuntu 20GB).
*   **Nhược điểm:** Bạn mua một Server vật lý chỉ để chạy 3 cái ứng dụng Python nhỏ gọn cộng lại khoảng 100MB, nhưng bạn lại phải tốn thêm tài nguyên lãng phí để gánh 3 cái hệ điều hành Ubuntu ảo chạy ngầm. Khởi động mất hàng phút, tốn kém RAM, chậm chạp và khó tự động hóa.

#### b. Container - Ảo hóa hệ điều hành
*   **Cơ chế:** Container bỏ hoàn toàn lớp Guest OS cồng kềnh. Nó dùng công nghệ của lõi Linux (như cgroups và namespaces) để bọc các ứng dụng lại. Tuyệt vời thay, hàng chục cái Container trên cùng một Server có thể **Dùng chung một hệ điều hành máy chủ (Share Host OS Kernel)**. 
*   **Ưu điểm:** Khởi động gần như ngay lập tức (vài giây), tiêu tốn đúng lượng RAM/CPU mà app cần (nghĩa là cực kỳ tiết kiệm, cực nhẹ). Một Server vốn chỉ trụ được 5 Máy Ảo nay có thể chạy được nghìn cái Container cùng lúc. Nó giải quyết tận gốc bài toán xung đột: App A và App B đang nằm trong 2 cái lồng sắt riêng biệt, không ai ảnh hưởng đến hệ sinh thái của ai dẫu khác nhau về phiên bản ngôn ngữ.

### 3. Định nghĩa Containerization (Khái quát hóa)

Từ sự ra đời của Container, chuẩn thực hành **Containerization** xuất hiện:

*   **Định nghĩa:** Đây là thao tác đóng gói toàn bộ mã nguồn ứng dụng (Source code) cùng với toàn bộ "hành lý" đi kèm của nó (Hành lý bao gồm: trình thông dịch runtime, các tool hệ thống, các thư viện dependencies, và cài đặt settings) thành một khối thống nhất (Image), đưa vào một hộp chứa cách ly (Container).
*   **Đặc tính vàng "Immutability" (Tính bất biến):** Nhờ việc gói mọi thứ vào một "kiện hàng" kín mít và tiêu chuẩn hóa, kiện hàng Container mang đi đâu (từ LapTop của Dev, lên server Staging/Test, cho đến đám mây AWS Cloud) thì nó cũng bung ra chạy y chang như nhau 100%. Xóa bỏ vĩnh viễn điệp khúc *"It works on my machine!"*. Nó mang lại nguyên lý "Build Once, Run Anywhere".

## II. Kiến trúc của Docker (Docker Architecture)

Trong thế giới Container, **Docker** hiện lên như một vị vua không ngai. Nó phổ biến đến mức người ta hay dùng nhầm lẫn ranh giới giữa hai từ "Container" và "Docker" như một khái niệm duy nhất. Thực chất, Docker chạy theo mô hình **Client - Server**. Để trả lời phỏng vấn trôi chảy (và thực sự biết cách debug lỗi khi hệ thống sập), bạn cần nắm vững 5 thành phần cốt lõi của Kiến trúc Docker:

### 1. Docker Daemon (Server/Engine)
*   **Là gì?** Nó chính là "Trái tim" hay bộ não của Docker. File này (thường có tên tiến trình là `dockerd`) chạy ngầm thầm lặng trên hệ điều hành máy chủ (Linux/Windows Server).
*   **Nhiệm vụ:** Lắng nghe các yêu cầu thông qua Docker Engine API và hì hục thực thi công việc nặng nhọc: Tải Image xuống, xây dựng (Build) Image mới, kích hoạt chạy Container, quản lý Mạng (Network ảo) và quản lý Ổ cứng (Volumes). Nếu quá trình `dockerd` này bị chết, toàn bộ hệ sinh thái Container trên máy chủ đó cũng "đóng băng".

### 2. Docker Client (CLI/Giao diện)
*   **Là gì?** Nó là công cụ bạn tương tác hàng ngày bằng màn hình đen Terminal. Khi bạn cài Docker lên máy tính, nó cấp cho bạn bộ lệnh dòng lệnh (Command Line Interface - CLI).
*   **Nhiệm vụ:** Hoàn toàn không tự xử lý gì cả. Mỗi khi bạn gõ một gõ lệnh quen thuộc như `docker build` hay `docker run`, Client chỉ đơn thuần là gửi cái lệnh đó qua sóng API (REST API) đến tay Docker Daemon để nó làm việc. Lợi thế của chia cắt Client-Server này là: Máy Client (Laptop của bạn) cấu hình cùi bắp vẫn có thể điều khiển một cục Daemon siêu to khổng lồ nằm trên Đám mây AWS qua cấu hình qua IP mạng `tcp://server-ip`.

### 3. Docker Images (Hình ảnh/Bản vẽ)
*   **Là gì?** Nó là một tệp (File) có tính chất **Chỉ Đọc (Read-only Template)**. Sinh viên thường rất khó mường tượng khái niệm này. Hãy nghĩ Image giống hệt như một file `.iso` dùng để cài đặt Windows hoặc file Rom `.gho` chuyên đi Ghost win dạo ngày xưa. Trong đời sống thực, nó giống như "Bản thiết kế kỹ thuật" của một ngôi nhà do kiến trúc sư vẽ sẵn.
*   **Cách hoạt động:** Image được xếp chồng lên nhau bằng hàng tá lớp (Layers). Ví dụ: Lớp đầu tiên là hệ điều hành Ubuntu $\rightarrow$ Lớp thứ hai cài đặt môi trường Python 3.9 $\rightarrow$ Lớp thứ ba sao chép code dự án của bạn vào $\rightarrow$ Lớp thứ tư chạy lệnh tải thư viện Pandas. Lưới Layer này làm cho Image cực kỳ linh hoạt và tái sử dụng lại phần nhân (Cache).

### 4. Docker Containers (Thùng chứa/Thực thể)
*   **Là gì?** Nếu Image là "Bản thiết kế kỹ thuật" nằm im lìm trên giấy, thì Container chính là **"Ngôi nhà bằng gạch thật" (Running instance)** đã được xây dựng trọn vẹn từ bản vẽ đó và đi vào hoạt động đón khách.
*   **Đặc điểm:** 
    *   Từ một cái Image, bạn có thể ấn nút lệnh (Tức là gọi hàm khởi tạo) để chạy ra 100 cái Container hoàn toàn độc lập với nhau.
    *   Khác với Image (Chỉ đọc), Container có thêm một "Lớp bọc trên cùng cho phép Đọc-Ghi (Read-Write Layer)". Nhờ thế thao tác file của app này sẽ không làm hỏng app khác. Khi tắt (Stop) và xóa (Remove) một Container, lớp dữ liệu trên cùng này cũng bốc hơi hoàn toàn vào quên lãng. (Sẽ nói ở phần Storage Volumes).

### 5. Docker Registries (Chợ Container)
*   **Là gì?** Code phần mềm lưu ở Git / GitHub. Còn các cục Image thiết kế ở trên sau khi đóng gói xong sẽ được đẩy cất giữ vào một cái Thư viện/Cửa hàng gọi là Registry.
*   **Phân loại:**
    *   *Public Registry (Kho công cộng):* Nổi tiếng nhất là **Docker Hub**. Bạn có thể lên đây tải sẵn hàng trăm nghìn mẫu Image miễn phí (Image chứa sẵn Postgres, Redash, Airflow, Spark...) về xài liền.
    *   *Private Registry (Kho nội bộ doanh nghiệp):* Trái tim của việc Deployment công ty lớn. Bạn không thể mang Code Source chứa mật khẩu Business của công ty công khai lên mạng cho cả thế giới tải về. Các công ty bắt buộc phải thiết lập các kho kín của riêng mình (Ví dụ: Amazon ECR, Google GCR, hoặc tự cài Harbor cục bộ) để bảo vệ bản quyền Image chặt chẽ. Kỹ sư phải có quyền Auth mới được Load Image về server chạy.

## III. Dockerfile - Công thức nấu ăn cho Image

Nếu coi Image là một món ăn hoàn chỉnh, thì **Dockerfile** chính là "Tờ công thức nấu ăn". Đó là một file văn bản thuần túy, không có đuôi mở rộng (chỉ viết đúng tên `Dockerfile`), chứa một loạt các chỉ thị (Instructions) đọc từ trên xuống dưới để Docker Engine biết cách nhào nặn ra cái Image cuối cùng.

### 1. Khái niệm Layer (Lớp) và Caching

Hệ thống hoạt động của Dockerfile không phải chạy một lèo như bash script, mà nó cực kỳ thông minh nhờ cơ chế **Layer**:
*   Mỗi một dòng lệnh (như `FROM`, `RUN`, `COPY`) trong Dockerfile sẽ tạo ra một **Layer (Lớp) mới** đắp chồng lên cái cũ.
*   **Cơ chế Cache:** Giả sử Dockerfile của bạn có 5 dòng. Hôm nay bạn build lần đầu mất 10 phút tải đủ thứ. Ngày mai, bạn chỉ sửa đúng 1 dòng Code của ứng dụng ở lớp thứ 4. Khi bạn gõ `docker build` lại, Docker sẽ tự động lấy từ Lớp 1 đến Lớp 3 trong bộ nhớ đệm (Cache) ra dùng luôn trong chớp mắt (0.1 giây), và nó chỉ bắt đầu chạy lại từ Lớp thứ 4 trở đi. Điều này làm cho việc Build Container mới cực kỳ thần tốc.

### 2. Các chỉ thị (Instructions) thường xem gặp nhất

Để viết được Dockerfile, bạn bắt buộc phải thuộc lòng 6 chỉ thị "trấn phái" sau:

*   **`FROM <image_gốc>`:** Bắt buộc phải là dòng đầu tiên. Nó định nghĩa nền móng (Base Image). Ví dụ: `FROM python:3.9-slim` (Lấy sẵn một nhân Linux siêu nhẹ đã cài sẵn Python 3.9).
*   **`WORKDIR <đường_dẫn>`:** Lệnh thâm nhập vào ổ đĩa bên trong Container và đặt con trỏ chuột tại đó. Giống như bạn gõ lệnh `cd /app` vậy. Từ các dòng lệnh dưới trở đi, mọi file copy vào sẽ đổ hết vào thư mục này.
*   **`COPY <source> <destination>`:** Lệnh thiết yếu nhất. Copy mã nguồn (source code), file cấu hình từ máy tính của bạn (bên ngoài) vào thẳng bụng của con Container (bên trong). Ví dụ: `COPY requirements.txt .`
*   **`RUN <câu_lệnh_bash>`:** Lệnh thực thi lúc *Đang Xây Dựng (Build-time)*. Thường dùng nhất để cài gói thư viện. Ví dụ: `RUN pip install -r requirements.txt`. Nó sẽ chạy lệnh này rồi chốt cứng (commit) kết quả vào một Layer mới.
*   **`EXPOSE <cổng_port>`:** Lệnh khai báo mồm (Document). Giúp báo hiệu cho ông Data Engineer khác đọc file và biết rằng: *"À, cái Container này khi chạy lên nó sẽ vểnh tai nghe ở Port 8080 nhé"*.
*   **`CMD` hoặc `ENTRYPOINT`:** Lệnh thực thi lúc *Khởi Chạy (Run-time)*. Hai thằng này quy định: Sau khi xây xong cái nhà, mở cửa ra thì ông chủ nhà sẽ làm hành động gì đầu tiên? Thường là lệnh bật máy chủ Web lên. Ví dụ: `CMD ["java", "-jar", "app.jar"]` hoặc `CMD ["python", "app.py"]`.

### 3. Ứng dụng thực tế: Viết Dockerfile

Dưới đây là công thức ráp một cái Dockerfile thực chiến điển hình cho một dự án Backend hoặc Data API bằng Python:

```dockerfile
# 1. Chọn hệ điều hành và ngôn ngữ lõi (Nền tảng)
FROM python:3.9-slim-buster

# 2. Định nghĩa thư mục làm việc bên trong lõi Container
WORKDIR /app

# 3. Copy file danh sách thư viện vào trước để tận dụng Cache
COPY requirements.txt .

# 4. Chạy lệnh cài đặt thư viện (Tạo ra một Layer bự chứa toàn bộ Libs)
RUN pip install --no-cache-dir -r requirements.txt

# 5. Copy toàn bộ Code dự án từ ngoài vào trong
COPY src/ ./src/

# 6. Khai báo Port API
EXPOSE 8000

# 7. Lệnh kích hoạt server khi Container start
CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 4. Tuyệt kỹ: Ngự khí với Multi-stage Build

Nếu bạn ghi điểm mức Junior với cái Dockerfile trên, thì **Multi-stage Build** chính là màn trình diễn của Senior Architect để "Ép cân" file Image.

*   **Vấn đề:** Giả sử dự án Java Spring Boot. Để build ra file `.jar`, bạn cần cài nguyên bộ **JDK** khổng lồ (tầm 400MB) và đống tool Maven. Nhưng sau khi có file `.jar` rồi, để chạy nó ở môi trường Production thì thực ra chỉ cần bộ **JRE** bé tẹo (khoảng 80MB). Nếu bạn ôm nguyên cái JDK 400MB nhồi chung vào Image Production thì quá nặng, lãng phí RAM, đẩy lên mạng chậm và dễ dính lỗ hổng bảo mật.
*   **Giải pháp Multi-stage:** Cho phép viết 2 hoặc nhiều khối `FROM` trong cùng 1 file. Khối trên dùng đồ xịn để xào nấu món ăn. Khối dưới chỉ lấy cái đĩa sạch, nhón lấy kết quả tinh túy của khối trên bày ra đĩa và vứt toàn bộ nồi niêu xoong chảo rác rưởi của khối trên đi.

**Ví dụ Code (Tự nén size từ 500MB xuống còn 100MB):**

```dockerfile
# --- Giai đoạn 1: Build Stage (Dùng lò nung khủng) ---
FROM maven:3.8.4-jdk-11 AS builder
WORKDIR /app
COPY pom.xml .
# Tải mồi các lib phụ thuộc
RUN mvn dependency:go-offline 
# Copy code thực tế vào
COPY src ./src
# Compile và nén ra cục file app.jar
RUN mvn clean package -DskipTests

# --- Giai đoạn 2: Production Stage (Siêu mỏng cánh) ---
FROM openjdk:11-jre-slim
WORKDIR /app
# Phép màu: Lấy cục file app.jar từ cái môi trường có tên 'builder' ở đoạn 1 ném sang đây
COPY --from=builder /app/target/app.jar .
EXPOSE 8080
# Chạy app
CMD ["java", "-jar", "app.jar"]
```
*Tương tự, các ông lớn như hệ thống API Gateway (Kong, OpenResty) cũng hay dùng trò này để build lại tầng Core C++ rồi ném kết quả file nhị phân (binary) .so /.dll sang một cái box Alpine siêu mỏng 5MB để chạy thực tế.*

## IV. Lưu trữ và Mạng lưới (Storage & Networking)

Kỹ năng viết Dockerfile là nền tảng, nhưng khả năng cấu hình Ổ cứng (Volume) và Mạng ảo (Networking) mới là thứ phân biệt một "Thợ gõ Docker lệnh" và một "Kiến trúc sư hệ thống thực thụ".

### 1. Docker Volumes (Bài toán Stateless vs. Stateful)

*   **Lời nguyền Vô trạng thái (Stateless):** Bản chất của một Container sinh ra là Vô trạng thái. Nó giống như bộ não của loài cá vàng. Mọi file bạn copy vào lệnh `COPY` hay mọi log file do ứng dụng sinh ra trong lúc chạy đều được lưu trên "**Layer Đọc-Ghi tạm thời**" nằm trên cùng. Nếu một ngày đẹp trời, Container bị Crash (Sập cục bộ) hoặc bị lệnh `docker rm` cưỡng chế xóa bỏ, toàn bộ lớp dữ liệu tạm thời đó sẽ bốc hơi sạch sẽ vĩnh viễn không thể cứu vãn.
*   **Thảm họa Database:** Điều này là hoàn toàn ổn nếu đóng gói Backend API (vì mất đi bật lại chạy bình thường). Nhưng hãy tưởng tượng bạn bọc một cục cơ sở dữ liệu (PostgreSQL) vào Container. Nếu quy luật "xóa bốc hơi" vẫn đúng, toàn bộ Data khách hàng của công ty sẽ chìm theo Container đó xuống đáy biển.
*   **Chiếc phao cứu sinh - Volumes (Gắn ổ đĩa):** Để giải quyết bài toán cần lưu trữ lâu dài (Stateful), Docker cung cấp công cụ có tên là Volume.
    *   **Cơ chế Mounting (Gắn kết):** Khái niệm này giống việc bạn cắm cái USB vào laptop con. Bạn cấu hình *"Khui một cái lỗ"* xuyên qua tường vỏ Container, rồi hàn dính thẳng ổ cứng đó vào một thư mục vật lý (Host Directory) nằm bên ngoài con Laptop của bạn.
    *   **Thực thi:** Lệnh dùng flag `-v`. Ví dụ chạy Postgres: `docker run -v /vi_tri_odung_thật/data_postgres : /var/lib/postgresql/data postgres`. Như vậy, dù cho ngày mai bạn vứt xóa thẳng con Container đó đi, thì dữ liệu vẫn nằm sờ sờ an toàn tuyệt đối ngoài máy tính thật của bạn. Hôm sau bật con Postgres mới lên, cắm lại cái "lỗ" (Mount Point) đó, dữ liệu cũ lại nguyên vẹn nề nếp như cũ.

### 2. Docker Networking (Mạng lưới ảo hóa)

Khái niệm tiếp theo: Làm sao một cụm 4 con Container (Một con Backend Spring Boot, một con Frontend React, một con cục Cache Redis, và một củ Database) tự trò chuyện kết nối được với nhau, mà Người ngoài không gọi cắc cớ vào được? Docker sinh ra công cụ quản lý Mạng riêng (Networking) có thể cấu hình rất uyển chuyển. Đặc biệt lưu ý 3 chế độ (Mode) sau:

*   **Bridge Network (Chế độ Cây cầu - Mặc định):**
    *   *Định vị:* Giống như một cái Mạng LAN nội bộ (Wifi riêng khép kín).
    *   *Đặc quyền DNS Resolution (Phân giải tên cực vi diệu):* Giả sử bạn tạo mới một Bridge Network tên `my-app-net`, nhốt ông Frontend và ông Backend vào chung. Lúc này, Backend có thể chọc vào Redis thông qua URL có chứa thẳng Tên của Container đó luôn: `redis://redis-container:6379`, thay vì phải nhớ và hard-code địa chỉ IP (như kiểu `172.18.0.3` vốn dĩ sẽ bị Docker xáo trộn thay IP liên tục sinh ra lỗi).
    *   *Tính biệt lập:* Hai con Container nằm trong cùng Bridge sẽ thầm thì được nhau. Kẻ nằm ngoài tự Bridge (hoặc ngoài Internet) thì hoàn toàn mù tịt. Để hở một cái cho khách xài, bạn dùng Flag `-p 8080:80` (Mở Lỗ Hổng qua vách tường đẩy dữ liệu từ cổng 8080 bên ngoài nối ống luồn vào cổng 80 bên trong).

*   **Host Network (Chế độ Chủ đè):**
    *   *Tính chất:* Xé bỏ hoàn toàn lớp vỏ ngụy trang mạng ảo. Container "ký sinh" trực tiếp lên Mạng thật ảo (Thẻ Network Interface Card - NIC) của máy chủ chứa nó.
    *   *Cái lợi:* Nhanh lướt vì loại bỏ sự cồng kềnh trích xuất giao thức NAT ẩn.
    *   *Cái nguy:* Đụng port rất dễ nổ. Ví dụ 2 Container Web cùng đòi vươn râu ra ngoài vách để giành giật chạy cổng Port 80 của máy vật lý thì chắc chắn chết. Giảm tính quy hoạch rõ ràng.

*   **None Network (Chế độ Xó tối):**
    *   *Tính chất:* Hoàn toàn phế bỏ Card Mạng, tắt tính năng kết nối dây cáp (Networking Stack). Container bị nhốt như Tôn Ngộ Không dưới núi Ngũ Hành Sơn, nội bất xuất ngoại bất nhập.
    *   *Công dụng cho ai?* Rất có ích đối với Container chạy Batch Jobs (Cron) siêu khép kín. Thí dụ file đó chỉ chạy ra tính toán cục bộ mã hóa/bẻ khóa file `.zip` cực mật, không cần lên mạng lấy data và cũng sợ bị hack từ xa thì gán chế độ này cực an toàn.

## V. Docker Compose - Điều phối cục bộ (Local Orchestration)

Bạn đã biết cách đóng gói 1 ứng dụng bằng Dockerfile, biết cách gắn Volume và nối Mạng. Nhưng một hệ thống Data Engineering thật sự KHÔNG BAO GIỜ chỉ có 1 Container. Nó là một "Đội quân" bao gồm: một con PostgreSQL, một con Kafka, một con MinIO, một con Debezium, một con Airflow... Nếu bạn phải mở Terminal và gõ lần lượt 10 lệnh `docker run` dài ngoằng cho từng con, truyền đúng tên Network, đúng Volume, đúng biến môi trường... thì chắc chắn sẽ sai và mất cả ngày. **Docker Compose** ra đời để xử lý nỗi đau này.

### 1. Khái niệm file `docker-compose.yml`

*   **Bản chất:** Docker Compose là một công cụ đi kèm khi cài Docker, cho phép bạn **mô tả toàn bộ kiến trúc hạ tầng** (gồm bao nhiêu Container, mỗi Container dùng Image gì, xài Network nào, gắn Volume nào, biến môi trường là gì) vào trong **duy nhất một file** có tên `docker-compose.yml` (Hoặc `compose.yaml`).
*   **Cú pháp YAML:** File này sử dụng định dạng YAML (Yet Another Markup Language) - một cú pháp cấu hình cực kỳ dễ đọc cho con người vì nó dùng khoảng trắng thụt đầu dòng (Indentation) thay vì các ký tự ngoặc nhọn rối mắt.
*   **Sức mạnh "Một chạm":** Sau khi bạn đã tỉ mỉ mô tả mọi thứ trong file `.yml`, chỉ cần một lệnh duy nhất là cả "Đội quân" Container đứng dậy đồng loạt:

```bash
# Khởi chạy MỌI dịch vụ được khai báo, chạy ngầm (detached mode)
docker-compose up -d

# Tắt và dọn dẹp MỌI dịch vụ
docker-compose down
```

### 2. Cấu trúc của file `docker-compose.yml`

Một file Compose chuẩn có 3 khu vực chính:
*   **`services:`** (Bắt buộc) - Liệt kê danh sách các Container (Dịch vụ). Mỗi dịch vụ tương ứng với một Container.
*   **`volumes:`** (Tùy chọn) - Khai báo các ổ đĩa dùng chung hoặc gắn kết dữ liệu lâu dài.
*   **`networks:`** (Tùy chọn) - Khai báo các Mạng ảo riêng biệt cho các nhóm dịch vụ.

### 3. Ví dụ thực chiến: Kiến trúc nền tảng dữ liệu (Data Platform)

Dưới đây là một file `docker-compose.yml` mẫu mô phỏng một nền tảng Data Engineering hoàn chỉnh ngay trên máy cá nhân. Đây là minh chứng rõ ràng nhất cho sức mạnh của Docker Compose:

```yaml
version: "3.8"

services:
  # === 1. CƠ SỞ DỮ LIỆU GỐC (Source Database) ===
  postgres:
    image: postgres:14-alpine
    container_name: source-postgres
    environment:
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: secret123
      POSTGRES_DB: ecommerce_db
    ports:
      - "5432:5432"
    volumes:
      # Gắn Volume để dữ liệu không bốc hơi khi xóa Container
      - pg_data:/var/lib/postgresql/data
    networks:
      - data-platform-net

  # === 2. BẮT SỰ KIỆN THAY ĐỔI (CDC với Debezium) ===
  debezium-connect:
    image: debezium/connect:2.4
    container_name: debezium-connector
    environment:
      BOOTSTRAP_SERVERS: kafka:9092
      GROUP_ID: 1
      CONFIG_STORAGE_TOPIC: connect_configs
      OFFSET_STORAGE_TOPIC: connect_offsets
    ports:
      - "8083:8083"
    depends_on:
      # Đảm bảo Kafka và Postgres phải SỐNG trước khi Debezium bật dậy
      - kafka
      - postgres
    networks:
      - data-platform-net

  # === 3. HÀNG ĐỢI SỰ KIỆN (Message Queue - Kafka) ===
  kafka:
    image: confluentinc/cp-kafka:7.5.0
    container_name: kafka-broker
    environment:
      KAFKA_BROKER_ID: 1
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
    ports:
      - "9092:9092"
    depends_on:
      - zookeeper
    networks:
      - data-platform-net

  zookeeper:
    image: confluentinc/cp-zookeeper:7.5.0
    container_name: zookeeper
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
    networks:
      - data-platform-net

  # === 4. KHO LƯU TRỮ ĐỐI TƯỢNG (Object Storage - MinIO thay S3) ===
  minio:
    image: minio/minio:latest
    container_name: minio-storage
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: minioadmin
      MINIO_ROOT_PASSWORD: minioadmin123
    ports:
      - "9000:9000"   # API S3-compatible
      - "9001:9001"   # Web Console UI
    volumes:
      - minio_data:/data
    networks:
      - data-platform-net

# --- Khai báo Volumes (Ổ cứng lâu dài) ---
volumes:
  pg_data:
  minio_data:

# --- Khai báo Network (Mạng LAN nội bộ) ---
networks:
  data-platform-net:
    driver: bridge
```

**Phân tích điểm đáng chú ý:**
*   **`depends_on`:** Chỉ thị quy định thứ tự khởi động. Debezium sẽ không dám bật dậy cho đến khi Kafka và Postgres báo hiệu "Sẵn sàng". Điều này ngăn chặn lỗi "Connection Refused" do các dịch vụ phụ thuộc chưa kịp khởi tạo.
*   **DNS nội bộ tự động:** Để ý rằng Debezium kết nối với Kafka bằng URL `kafka:9092`. Đây chính là đặc quyền DNS Resolution của Bridge Network mà ta đã đề cập ở phần IV. Không cần IP tĩnh, chỉ cần gọi Tên dịch vụ là thông.
*   **Volumes riêng biệt:** `pg_data` và `minio_data` được định nghĩa ở cuối cùng, đảm bảo dữ liệu PostgreSQL và MinIO vẫn tồn tại vĩnh viễn dù cho bạn gõ `docker-compose down` xóa toàn bộ Container.
*   **Chỉ một Network duy nhất:** Tất cả dịch vụ cùng nằm trong `data-platform-net`, tạo nên một cụm mạng ảo khép kín hoàn hảo. Máy tính của bạn vừa biến thành một Data Center thu nhỏ chỉ bằng một lệnh `docker-compose up -d`.

## VI. Góc độ Phỏng vấn (Interview Scenarios for Junior)

Docker là kỹ năng "Vé vào cửa" bắt buộc cho mọi vị trí Data/Backend Engineer hiện đại. Dưới đây là 3 tình huống rất phổ biến mà các Interviewer hay đem ra tra hỏi ứng viên để đánh giá mức độ thành thạo Docker ở cấp Junior:

### Tình huống 1: Tối ưu dung lượng Image (Optimization)

*   **Câu hỏi:** *"Image Docker của dự án team bạn nặng tới 1.2GB. Bạn có những cách tiếp cận nào để ép cân nó xuống dưới 150MB?"*
*   **Gợi ý trả lời ghi điểm:** Ứng viên nên liệt kê được ít nhất 3 mũi tấn công sau:
    *   **Mũi 1 - Thay đổi Base Image:** Thay vì dùng Image nền gốc chuẩn, ví dụ `FROM python:3.9` (nặng ~900MB vì bao gồm cả đống tool Linux không cần thiết), hãy chuyển qua dùng biến thể siêu mỏng `FROM python:3.9-alpine` hoặc `FROM python:3.9-slim` (chỉ nặng ~40-120MB). Alpine Linux là bản phân phối cực nhẹ chỉ ~5MB, được cộng đồng Docker chuộng dùng nhất cho Production.
    *   **Mũi 2 - Multi-stage Build:** Đây chính là "Tuyệt kỹ" đã trình bày ở phần III.4. Tách giai đoạn Compile (cần JDK/Maven nặng) ra khỏi giai đoạn Run (chỉ cần JRE nhẹ). Image cuối cùng chỉ chứa kết quả tinh túy (file `.jar`, file nhị phân `.bin`) mà vứt hết rác rưởi của quá trình xây dựng.
    *   **Mũi 3 - Gộp lệnh RUN, giảm Layer:** Mỗi lệnh `RUN` tạo ra một Layer mới. Nếu bạn viết tách:
        ```dockerfile
        RUN apt-get update
        RUN apt-get install -y curl
        RUN apt-get clean
        ```
        Thì sẽ tạo ra 3 Layers rời rạc, trong đó Layer 1 và 2 vẫn giữ lại "rác" cache của `apt-get`. Hãy gộp lại thành một dòng duy nhất bằng toán tử `&&`:
        ```dockerfile
        RUN apt-get update && apt-get install -y curl && apt-get clean && rm -rf /var/lib/apt/lists/*
        ```
        Như vậy chỉ tạo ra 1 Layer duy nhất, và rác được dọn sạch ngay trong chính Layer đó.
    *   **Mũi 4 - File `.dockerignore`:** Giống `.gitignore`, file này nói với Docker: *"Đừng copy mấy cục thư mục rác này (như `node_modules/`, `.git/`, `__pycache__/`) vào trong Image nhé!"*. Việc này vừa giảm kích thước Image vừa tăng tốc build vì Docker không phải quét toàn bộ thư mục dự án.

### Tình huống 2: Xử lý lỗi Container (Troubleshooting)

*   **Câu hỏi:** *"Ứng dụng được deploy bằng Docker. Container vừa bật lên được 2 giây thì lập tức tự tắt (trạng thái Exited hoặc CrashLoopBackOff trong Kubernetes). Bạn sẽ dùng đội hình lệnh nào để xác định nguyên nhân?"*
*   **Gợi ý trả lời ghi điểm:** Đây là bài kiểm tra kỹ năng "Thám tử Debug" quan trọng bậc nhất. Hãy trình bày theo trình tự từ nhẹ đến nặng:
    *   **Bước 1 - Xem nhật ký (`docker logs`):**
        ```bash
        docker logs <tên_container> --tail 100
        ```
        Đây là hành động ĐẦU TIÊN phải làm. 90% trường hợp, dòng lỗi cuối cùng in ra trên màn hình sẽ nói cho bạn biết vấn đề (Ví dụ: `FileNotFoundError`, `Connection refused to postgres:5432`...).
    *   **Bước 2 - Soi ruột (`docker inspect`):**
        ```bash
        docker inspect <tên_container>
        ```
        Lệnh này sẽ "bổ tung" con Container ra cho bạn xem toàn bộ thông tin cấu hình bên trong dưới dạng JSON: Biến môi trường có đúng chưa? Network có gắn đúng chưa? Volume mount đúng đường dẫn chưa? Exit Code là bao nhiêu?
    *   **Bước 3 - Chui vào bên trong (`docker exec` hoặc Override Entrypoint):**
        Nếu 2 bước trên vẫn chưa tìm ra, bạn cần "mở nắp hầm" chui vào bên trong Container để kiểm tra thủ công bằng cách ghi đè lệnh khởi chạy:
        ```bash
        # Chạy Container nhưng thay vì bật app, mở shell ra cho tôi gõ lệnh
        docker run -it --entrypoint /bin/sh <tên_image>
        ```
        Từ bên trong, bạn có thể kiểm tra: File cấu hình có tồn tại không (`ls -la`), thư viện có đủ không (`pip list`), có ping được đến Database không (`ping postgres`)...

### Tình huống 3: Bảo mật Container (Security)

*   **Câu hỏi:** *"Tại sao không nên chạy ứng dụng bên trong Container bằng quyền Root User mặc định? Và bạn sẽ khắc phục điều này trong Dockerfile như thế nào?"*
*   **Gợi ý trả lời ghi điểm:** Đây là câu hỏi đo lường tư duy bảo mật (Security mindset) của ứng viên.
    *   **Rủi ro chết người:** Mặc định, tiến trình bên trong Container chạy bằng quyền `root` (Quyền tối thượng, tương đương God Mode). Nếu Hacker tìm được một lỗ hổng trong ứng dụng web của bạn (Ví dụ: lỗi Remote Code Execution) và "chui" được vào bên trong Container, hắn sẽ có toàn quyền điều khiển. Tệ hơn nữa, trong một số cấu hình Docker lỏng lẻo (chạy Container ở Privileged mode), Hacker có thể **"vượt ngục" (Container Escape)** leo từ Container ra máy chủ (Host) thật ngoài kia và chiếm quyền kiểm soát toàn bộ Server.
    *   **Nguyên tắc Đặc quyền Tối thiểu (Principle of Least Privilege):** Ứng dụng chỉ nên có đúng mức quyền hạn tối thiểu mà nó cần để hoạt động.
    *   **Cách khắc phục trong Dockerfile:** Thêm chỉ thị `USER` để tạo và chuyển sang một user thường (Non-root user):
        ```dockerfile
        FROM python:3.9-slim

        # Tạo một user mới tên 'appuser' không có quyền root
        RUN addgroup --system appuser && adduser --system --ingroup appuser appuser

        WORKDIR /app
        COPY . .
        RUN pip install -r requirements.txt

        # Chuyển sang user thường TRƯỚC KHI chạy ứng dụng
        USER appuser

        CMD ["python", "app.py"]
        ```
        Từ dòng `USER appuser` trở xuống, mọi lệnh bên trong Container đều chạy bằng quyền của `appuser` thay vì `root`. Kể cả Hacker có chui vào được, hắn cũng chỉ là một "Dân thường" bị hạn chế quyền rất chặt, không thể cài phần mềm độc hại hay sửa file hệ thống.
